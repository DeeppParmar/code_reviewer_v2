This document details the chronological testing progression, architectural changes, the reasoning behind each update, and their subsequent impact on model performance scoring across the Code Review OpenEnv.

--------------------------------------------------------------------------------
Phase 1: Baseline Architecture & Flawed Ceiling 
--------------------------------------------------------------------------------
[State & Problem]
- Initially, the environment suffered from a mathematical bug where F1 scores would artificially hit a ceiling of 0.999.
- The "done" reward was heavily coupled within the F1 logic, and False Positives (models complaining about things that weren't bugs) were not being properly penalized mathematically.

[Changes Made]
- Overhauled `base_grader.py` using a weighted F1 formula based on severity multipliers.
- Mathematical logs were injected so that True Positives, False Positives (weighted), and recall constraints were explicitly logged line-by-line during grading.
- Implemented environment reward clamping [0.0, 1.0] to satisfy the OpenEnv specification strictly.

[Impact]
- Grading became brutal, deterministic, and highly transparent. Ceiling inflation vanished, and models that spammed false positives (like early Llama generations) were mathematically crushed to scores around 0.10 - 0.20 instead of scoring high.

Phase 4: Max Token Truncation & Network Timeout Overhaul (logs-3.txt) [Latest Run - Time: ~2026-04-11T13:17+05:30]
--------------------------------------------------------------------------------
[State & Problem]
- The user masterfully identified two hidden bugs in `inference.py`: 
  1) `_get_max_tokens` was explicitly capping `gemma` at 512 max output tokens. This resulted in Gemma getting truncated mid-JSON generation, throwing parser errors and crashing.
  2) OpenRouter was hanging indefinitely. The Python `openai` socket was waiting up to 600s, crippling iteration speed when hitting endpoints like Mixtral.

[Changes Made]
- Overhauled `_get_max_tokens` so that `27b` inherently receives a 1024 token limit buffer.
- Added a hard socket timeout constraint explicitly within the Python OpenAI client definition: `llm = OpenAI(..., timeout=120.0)`
- Fired off with a fresh OpenRouter API key to isolate network limits.
